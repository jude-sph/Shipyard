You are generating Arcadia Logical Architecture (LA) elements for the Capella modeling tool.

PURPOSE OF THIS STAGE:
The Logical Architecture describes the major elements defining the notional solution in a synthetic manner. It opens the system black box into logical subsystems (components), allocates functions to those components, and defines the notional functional exchanges and component exchanges between them. Technology choices are NOT made here -- this is a technology-independent decomposition.

ARCADIA ELEMENT DEFINITIONS:

1. Logical Components (components)
   A component performing functions (behavioral component) is a constituent of the system contributing to its behaviour and properties, by implementing functions interacting with other components.
   Fields: id, name, functions (list of logical function IDs allocated to this component)

2. Logical Functions (functions)
   A notional function is an action, operation, or service fulfilled by the system, or by an actor interacting with the system.
   Fields: id, name, component (single component ID to which this function is allocated)

3. Notional Solution Capabilities (notional_capabilities)
   A notional solution capability is the way the system provides a service that supports the achievement of a mission, notably specified capabilities.
   Fields: id, name, involved_functions (list of function IDs), involved_chains (list of functional chain IDs)

4. Notional Functional Chains (functional_chains)
   A notional functional chain describes the behavior of the system in a given context. It is a logical organization of functions and functional exchanges to fulfil a Capability.
   Fields: id, name, function_refs (list of function IDs in the chain), exchange_refs (list of functional exchange IDs in the chain)

5. Notional Scenarios (notional_scenarios)
   A notional scenario describes the behavior of the system in a given context. It is a time-ordered set of functional exchanges between functions, or between the system and external actors, to fulfil a Capability.
   Fields: id, name, steps (list of step objects)
   Each step: from_entity (function ID or component ID), to_entity (function ID or component ID), message (string), sequence (integer)

6. Notional Functional Exchanges (functional_exchanges)
   A notional functional exchange is an interaction from a source function likely to deliver exchange items to another function (information, signal, material, torque, etc.).
   Fields: id, name, source_function (function ID), target_function (function ID), exchanged_items (list of exchanged item IDs)

7. Notional Exchanged Items (exchanged_items)
   A notional exchange contents describes what is exchanged by the system, system components, actors, and functions allocated to each of them.
   Fields: id, name, description (string)

8. Notional Component Exchanges (component_exchanges)
   A (behavioral) component exchange is an interaction between two components likely to exchange some items (information, signal, material, torque, etc.). It realises functional exchanges between functions allocated to source and target components.
   Fields: id, name, source_component (component ID), target_component (component ID)

9. Notional Interfaces (interfaces)
   Interfaces group various exchange items, and are allocated to behavioural component exchanges.
   Fields: id, name, component_exchange_ref (single component exchange ID), exchange_items (list of exchanged item IDs)

10. Notional Modes & States (modes_and_states)
    An automaton describes how modes and states are linked by transitions; it is linked to the appropriate object (system, component, etc.).
    Fields: id, name, type ("Mode" or "State"), transitions (list of objects with "from", "to", "trigger" keys)
