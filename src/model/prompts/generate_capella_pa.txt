You are generating Arcadia Physical Architecture (PA) elements for the Capella modeling tool.

PURPOSE OF THIS STAGE:
The Physical Architecture describes elements defining the finalised solution, at a level of detail sufficient to develop, integrate, and verify it. It deploys logical components onto real hardware and software nodes, defines the physical links between them, and makes concrete technology choices. This is the stage where the solution becomes implementable.

ARCADIA ELEMENT DEFINITIONS:

1. Physical Components (components)
   A component performing functions (behavioral component) is a constituent of the system contributing to its behaviour and properties, by implementing functions interacting with other components. In the Physical Architecture, these are actual hardware units, software components, or network elements.
   Fields: id, name, type ("HW_Node", "SW_Component", or "Network"), logical_components (list of logical component IDs from LA that this physical component realises)

2. Design Functions (functions)
   A design function is an action, operation, or service fulfilled by the system, or by an actor interacting with the system. These are physically hosted on a specific component.
   Fields: id, name, physical_component (single physical component ID hosting this function)

3. Physical Component Links (links)
   A physical link is a communication means between two physical hosting components (cables, buses, network connections, etc.).
   Fields: id, name, source (physical component ID), target (physical component ID)

4. Designed Solution Capabilities (designed_capabilities)
   A designed solution capability is the way the system provides a service that supports the achievement of a mission, notably specified capabilities.
   Fields: id, name, involved_functions (list of function IDs), involved_chains (list of functional chain IDs)

5. Design Functional Chains (functional_chains)
   A design functional chain describes the behavior of the system in a given context. It is a logical organization of functions and functional exchanges to fulfil a Capability.
   Fields: id, name, function_refs (list of function IDs), exchange_refs (list of functional exchange IDs)

6. Design Scenarios (design_scenarios)
   A design scenario describes the behavior of the system in a given context. It is a time-ordered set of exchanges between components, or between the system and external actors, to fulfil a Capability.
   Fields: id, name, steps (list of step objects)
   Each step: from_entity (component ID or function ID), to_entity (component ID or function ID), message (string), sequence (integer)

7. Design Functional Exchanges (functional_exchanges)
   A design functional exchange is an interaction from a source function likely to deliver exchange items to another function (information, signal, material, torque, etc.).
   Fields: id, name, source_function (function ID), target_function (function ID), exchanged_items (list of exchanged item IDs)

8. Design Exchanged Items (exchanged_items)
   A design exchange contents describes what is exchanged by the system, system components, actors, and functions allocated to each of them.
   Fields: id, name, description (string)

9. Component Exchanges (component_exchanges)
   A (behavioral) component exchange is an interaction between two components likely to exchange some items (information, signal, material, torque, etc.). It realises functional exchanges between functions allocated to source and target components.
   Fields: id, name, source_component (physical component ID), target_component (physical component ID)

10. Physical Hosting Components (hosting_components)
    A physical hosting component is a component hosting behavioural components, delivering them required resources and support for interactions or communications between them.
    Fields: id, name, hosted_components (list of physical component IDs that are hosted)

11. Designed Interfaces (interfaces)
    Interfaces group various exchange items, and are allocated to behavioural component exchanges.
    Fields: id, name, component_exchange_ref (single component exchange ID), exchange_items (list of exchanged item IDs)

12. Design Modes & States (modes_and_states)
    An automaton describes how modes and states are linked by transitions; it is linked to the appropriate object (system, component, etc.).
    Fields: id, name, type ("Mode" or "State"), transitions (list of objects with "from", "to", "trigger" keys)
